# Agent Stream Protocol

Status: current native TypeScript implementation + target direction

This document describes the Aithru Agent SSE stream protocol — both what is currently
implemented and the target state where they differ.

The stream is a structured, replayable, append-only run event stream that drives chat
UI, run timeline, workspace, artifact, approval, debug trace, and audit projections.

## One-line definition

```txt
Agent Stream = Chat output + Harness runtime events + Tool/todo/artifact/approval events
as one structured SSE event stream.
```

## Transport

V1 uses Server-Sent Events (SSE) over HTTP `GET`:

```
GET /api/runs/{run_id}/stream?after_sequence=0&follow=true&poll_interval_seconds=0.01
GET /api/threads/{thread_id}/runs/{run_id}/stream  (thread-scoped variant)
```

Live run creation via POST also returns an SSE stream:

```
POST /api/runs/stream
POST /api/threads/{thread_id}/runs/stream
```

All respond with `Content-Type: text/event-stream`.

Reverse operations are REST:

```
POST /api/runs/{run_id}/cancel
POST /api/runs/{run_id}/resume
POST /api/runs/{run_id}/input
POST /api/approvals/{approval_id}/resolve
```

## SSE wire format

Generated by `formatSseEvent` in `backend/packages/stream/src/sse.ts`.

```
id: {event.id}
event: {event.type}
data: {compact JSON of entire AgentStreamEvent}
```

Example:

```
id: run-001:3
event: message.delta
data: {"id":"run-001:3","run_id":"run-001","thread_id":null,"sequence":3,"timestamp":"2026-06-24T10:30:00Z","type":"message.delta","source":{"kind":"model","id":"gpt-4o"},"visibility":"user","redaction":"none","summary":null,"payload":{"message_id":"msg-1","delta":"Hello"}}
```

Frames are separated by `\n\n`. Each frame contains `data:` lines carrying the
JSON-serialized event. The JSON uses compact separators (`separators=(',', ':')`).

> **Target:** Add heartbeat frames for long idle periods. Add a `run.terminal` event
> sent before stream close. Both are not yet implemented.

## Event envelope

Defined by the TypeBox schema in
`backend/packages/contracts/src/schemas.ts` and the TypeScript type exported
from `backend/packages/contracts/src/types.ts`.

```ts
type AgentStreamEvent = {
  id: string;            // "{run_id}:{sequence}", globally unique
  run_id: string;
  thread_id: string | null;
  sequence: number;       // strictly increasing within a run, 1-based
  timestamp: string;      // ISO 8601 UTC, e.g. "2026-06-24T10:30:00Z"
  type: string;           // event type (see below)
  source: {
    kind: string;
    id?: string;
    name?: string;
  };
  visibility: "user" | "debug" | "audit";   // default "user"
  redaction: "none" | "partial" | "full";   // default "none"
  summary: string | null;
  payload: Record<string, unknown>;
};
```

Rules:

- `sequence` is strictly increasing per run.
- `id` is globally unique.
- `timestamp` is server-generated at event write time.
- `visibility=user` is safe for normal UI; `debug` may include technical details
  but no secrets; `audit` carries governance metadata.
- `redaction` reflects whether sensitive fields in payload were redacted.
  `FULL` takes precedence over `PARTIAL` over `NONE` when combining explicit
  and detected redaction.

## Event persistence

Every event is written to the active event store before being broadcast to SSE
clients. The native backend supports in-memory storage for tests and local
development plus SQLite-backed storage for durable local runs. Events are
indexed by `run_id` and `sequence`.

```
AgentEventWriter.write(type, payload, ...)
  -> AgentStore.appendEvent(runId, event)
  -> formatted by formatSseEvent(event)
  -> yielded to SSE clients
```

> **Target:** Replace in-memory store with a durable `EventStore` backend.
> Persist-before-publish is already enforced.

## Stream lifecycle

The run stream route in `backend/apps/api/src/routes/runs.ts`:

1. Reads historical events after `after_sequence`.
2. For each batch, formats and yields `formatSseEvent(event)`.
3. If a terminal event type is seen, closes the stream.
4. Else polls with `poll_interval_seconds` delay until:
   - a terminal event appears;
   - or the run is not found.
5. While idle, yields SSE comment keepalives (`: keepalive\n\n`).

`timeout_seconds` is optional and defaults to no timeout. It is a diagnostic or
test guard, not the normal lifecycle signal.

Terminal event types:

```txt
run.completed
run.failed
run.cancelled
```

For live streams (`_run_live_stream_response`), also launches a background
`worker.drain(limit=1)` to eagerly process the run.

## Frontend consumption

The frontend parses SSE with `fetch` + `ReadableStream` reader in
`[client.ts](frontend/src/lib/api/client.ts#L111-L149)`:

1. `fetch(url, { headers: { Accept: "text/event-stream" } })`
2. Read chunks from `response.body.getReader()`
3. Split on `\n\n` to extract frames
4. Extract `data:` lines; join multi-line data
5. If data is `[DONE]`, close
6. `JSON.parse(data)` and call `onEvent`

Events are then reduced through `reduceEvent` in
`[useRunStream.ts](frontend/src/features/chat/useRunStream.ts#L114-L337)` into a
`RunStreamState` that drives chat UI:

```ts
interface RunStreamState {
  status: "idle" | "running" | "waiting_input" | "waiting_approval" | "paused" | "completed" | "failed" | "cancelled";
  messages: ChatMessage[];
  toolCalls: ToolCallEntry[];
  todos: TodoEntry[];
  inlineRequests: InlineRequest[];
  tokenUsage?: { input?: number; output?: number; total?: number };
  error?: string;
}
```

### Message events

| type | status | payload |
|------|--------|---------|
| `message.created` | ✅ | `message_id`, `role` ("user" \| "assistant"), `content` |
| `message.delta` | ✅ | `message_id`, `delta` (incremental text) |
| `message.completed` | ✅ | `message_id`, `content` (final), optional `thread_message_id` |
| `message.failed` | ✅ | `message_id`, `content`, `error` |

`message.delta` is the primary user-facing assistant text stream.
`message.completed` carries the full assembled content.
For threaded assistant runs, `message_id` remains the run-local stream message id;
`thread_message_id` points at the persisted assistant thread message.
The frontend accumulates `delta` chunks into a growing `ChatMessage.content`,
then finalizes on `message.completed`.

### Tool events

| type | status | payload |
|------|--------|---------|
| `tool.proposed` | ✅ | `tool_call_id`, `tool_name`, `input`, `risk_level`, `approval_policy` |
| `tool.started` | ✅ | `tool_call_id`, `tool_name` |
| `tool.completed` | ✅ | `tool_call_id`, `tool_name`, `output`, `result`, `capability_key` |
| `tool.failed` | ✅ | `tool_call_id`, `tool_name`, `error`, `reason` |
| `tool.denied` | ✅ | `tool_call_id`, `tool_name`, `reason` |

Tool event ordering:

**Safe/read tool (no approval):**
```
tool.proposed  →  tool.started  →  tool.completed
```

**Dangerous/write tool (approval required):**
```
tool.proposed  →  approval.requested  →  run.paused
  →  [user resolves]  →  approval.resolved  →  run.resumed
  →  tool.started  →  tool.completed
```

`tool.completed`, `tool.failed`, and `tool.denied` events carry governance payloads
including `authorization_decision` and capability `audit` metadata (redacted
before stream publication).

## Redaction

Event payloads are automatically scanned for sensitive keys before storage and
publishing. The redaction logic in `backend/packages/stream/src/redaction.ts`
handles:

- Direct matches: `api_key`, `token`, `secret`, `password`, `credential`, `authorization`, etc.
- Suffix patterns: `*_token`, `*_secret`, `*_password`
- Nested keys in dicts and lists (recursive)
- Normalizes key names (lowercase, dash/space to underscore)

Sensitive values are replaced with `[REDACTED]` and the event's `redaction`
field is set to `partial` (or `full` if explicitly requested).

## Reconnect and replay

Clients reconnect by specifying `after_sequence`:

```
GET /api/runs/{run_id}/stream?after_sequence=42
```

When `follow=false` (default), the server returns all events after `after_sequence`
as a single SSE response and closes.

When `follow=true`, the server sends historical events then polls for new ones
until the run emits `run.completed`, `run.failed`, or `run.cancelled`. Idle
streams send keepalive comments; clients should treat missing keepalive bytes as
a connection error and reconnect with the latest `after_sequence`.

## Filtering

> Not yet implemented. The target supports `?visibility=user` and `?types=message.delta,artifact.created`.

## Projections

One event stream drives multiple UI projections:

| Projection | Consumes |
|------------|----------|
| Chat | `message.*`, `run.*`, `approval.*`, `input.*` |
| Run timeline / activity | `run.*`, `todo.*`, `tool.*`, `subagent.*` |
| Workspace | `workspace.*` |
| Artifact panel | `artifact.*` |
| Approval panel | `approval.*`, `tool.*` |
| Debug trace | all events |
| Audit | `visibility=audit` events |
