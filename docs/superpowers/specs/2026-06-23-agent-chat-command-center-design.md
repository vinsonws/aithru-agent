# Agent Chat Command Center Design

Date: 2026-06-23

## Goal

Redesign the Aithru Agent chat surface into a Cursor/Codex-style agent chat application.
The chat should remain the primary workspace, while task execution, approvals, files,
artifacts, usage, and traceability are always close at hand.

This design keeps Aithru Agent as an AI harness. It does not introduce workflow graph
editing, persisted workflow definitions, or agent-owned workflow semantics.

## Selected Direction

Use a three-column Command Center layout:

1. Left column: conversation inbox.
2. Center column: readable chat transcript and composer.
3. Right column: Activity-first run companion.

The chosen default for the right column is Activity First. Files, approvals, and trace
remain tabs within the same panel and become prominent only when they need attention.

## Product Principles

- Conversation first: the user should feel they are chatting with an agent, not operating
  an admin console.
- Execution visible: the agent's plan, current step, tool activity, approvals, and final
  result should be visible without burying the chat.
- Detail on demand: raw trace and low-level events should be available but collapsed by
  default.
- Actionable states: waiting input, approval needed, failed tool, completed artifact, and
  running status should be visually obvious.
- Stable layout: the app should not resize or jump as messages, runs, tool calls, or
  artifacts appear.

## Desktop Layout

### Left Conversation Inbox

The existing sidebar becomes a conversation inbox rather than a generic manager list.

It should include:

- app identity and a new chat button;
- search for conversations;
- pinned and recent groups;
- each thread row with title, latest status, attention marker, progress hint, and relative
  time;
- bottom links for Skills, Memory, Settings, and other management surfaces.

Thread rows should show agent status in plain language:

- Running;
- Awaiting input;
- Approval needed;
- Failed;
- Completed;
- Artifacts created.

The collapsed sidebar remains icon-only, but should preserve unread/attention badges.

### Center Chat Surface

The center panel is the primary working area.

Header content:

- editable thread title;
- current run identifier or compact run context;
- run status badge;
- selected model profile;
- optional quick access to share/export or inspect actions.

Message transcript:

- user messages use compact right-aligned bubbles;
- assistant messages use readable left-aligned surfaces;
- markdown remains first-class for code blocks, lists, links, and tables;
- streaming should show an inline cursor and keep auto-scroll stable;
- failed runs should produce a compact error card with a path to trace details;
- waiting-for-input should appear once, inline, with a clear reply action.

Agent activity inside the transcript:

- plan cards, tool summaries, approval prompts, and artifact summaries can appear inline;
- raw tool payloads and trace-level details stay collapsed;
- long activity should summarize into a compact card with "view details" opening the right
  companion.

### Composer

The composer should feel like a full chat application input, not a form.

Default visible controls:

- multiline text input;
- attach file button;
- send button;
- stop button when a run is active;
- compact Mode, Model, and Skill selectors.

Behavior:

- Enter sends, Shift+Enter inserts a newline;
- the text area grows up to a fixed max height and then scrolls internally;
- model and skill settings should remain available but visually secondary;
- advanced configuration belongs in settings, not the default composer surface;
- file upload hooks should connect to workspace files when that backend path is ready.

## Right Run Companion

The right panel is renamed conceptually from Inspection to Run Companion.

Default tab: Activity.

Tabs:

- Activity;
- Files;
- Approvals;
- Trace.

Optional secondary areas can remain available where useful:

- Artifacts;
- Memory;
- Usage.

### Activity Tab

The Activity tab is the everyday view.

It should show:

- current run status and elapsed time;
- progress through agent steps or todos when available;
- current action;
- approvals or input requests that need attention;
- latest tool activity summarized in human language;
- output summary;
- token/request usage.

Activity rows should be grouped into:

- completed;
- current;
- waiting;
- failed;
- next.

The current step should be visually stronger than historical steps.

### Files Tab

The Files tab should show workspace impact.

It should include:

- modified, created, deleted, and read files;
- artifacts generated by the run;
- file preview or diff entry points;
- empty state when no workspace changes exist.

This tab should show a badge when artifacts or file changes appear.

### Approvals Tab

The Approvals tab should show only actionable approval state by default.

It should include:

- pending approval cards;
- requested action summary;
- risk or scope;
- approve/deny controls;
- resolved approvals history collapsed below.

The app may auto-focus this tab when user action is required.

### Trace Tab

The Trace tab is for debugging and audit.

It should include:

- event stream;
- span tree;
- model usage events;
- tool calls and outcomes;
- failure diagnostics.

Trace should not be the default tab for normal chat.

## Mobile and Narrow Layout

On smaller screens:

- the left conversation inbox collapses into a drawer;
- the right run companion collapses into a bottom sheet or drawer;
- the center chat remains primary;
- Activity alerts surface as compact inline cards when the companion is hidden;
- composer controls move into a compact toolbar/menu.

The mobile experience should still allow users to:

- create a new chat;
- switch conversations;
- send and stop a run;
- answer input/approval prompts;
- inspect files or trace when needed.

## State Model

The UI should continue to map to existing backend concepts:

- Agent Thread maps to conversation;
- Agent Run maps to an execution instance inside a thread;
- Agent Message maps to transcript entries;
- AgentStreamEvent maps to streaming message, activity, tool, trace, and usage updates;
- Workspace and Artifact maps to files and outputs;
- Approval and input events map to actionable cards.

No new workflow definition model is introduced.

## Data Flow

1. User sends a message from the composer.
2. Frontend creates a run with the active thread, selected model profile, selected skill,
   persisted goal message, and configured scopes.
3. Frontend follows the run stream.
4. Stream events update:
   - transcript messages;
   - inline agent activity cards;
   - right-panel Activity timeline;
   - Files, Approvals, and Trace tabs;
   - thread inbox status and badges.
5. Completed runs update the final assistant message, result summary, usage, artifacts,
   and thread latest activity.

## Component Design

Recommended frontend component boundaries:

- `ConversationInbox`: left rail, search, grouping, thread row states.
- `ConversationHeader`: title editing, run/model status, compact actions.
- `ChatTranscript`: message list, streaming, error cards, inline activity cards.
- `ChatMessage`: user and assistant message rendering.
- `AgentActivityCard`: compact plan/tool/approval/artifact summaries inside chat.
- `ChatComposer`: input, send/stop, attachment, mode/model/skill controls.
- `RunCompanion`: right panel shell, collapse behavior, tabs.
- `RunActivityTab`: progress, current step, timeline, usage summary.
- `RunFilesTab`: workspace files, diffs, artifacts.
- `RunApprovalsTab`: pending and resolved approvals.
- `RunTraceTab`: event and span details.

Existing components can be evolved rather than replaced wholesale:

- `Sidebar` becomes the conversation inbox.
- `ConversationPage` owns the three-column chat layout.
- `ChatPanel` becomes the transcript surface.
- `ChatComposer` keeps run creation but moves selectors into compact controls.
- `InspectionPanel` becomes the run companion shell.

## Visual Direction

Use a restrained, product-focused interface:

- white main surfaces on a pale neutral app canvas;
- subtle borders and low shadows;
- indigo primary accent for user action and selected state;
- green for completed state;
- amber for waiting/approval state;
- red for failure state;
- compact radii, around 8-12px depending on component size;
- no decorative gradient blobs or marketing-style hero treatment.

The app should feel calm, capable, and operational.

## Error Handling

Failure states should appear in both the transcript and Activity tab:

- transcript: concise error card with human-readable failure;
- Activity: failed step with diagnostic summary;
- Trace: detailed event/span payload for debugging.

For validation and configuration errors, prefer actionable text:

- what failed;
- which configuration or field caused it;
- what the user can change next.

Repeated inline input or approval cards should be de-duplicated per active run state.

## Accessibility

- All icon-only controls require accessible labels and tooltips.
- Keyboard users must be able to navigate thread list, tabs, composer, send/stop, and
  approval controls.
- Focus should move predictably after sending, stopping, approving, or replying.
- Status changes should be announced where practical without noisy repetition.
- Text must not overflow controls on narrow screens.

## Testing

Implementation should include focused tests for:

- run stream event projection into transcript and Activity tab;
- no duplicate waiting input cards;
- composer send/stop behavior;
- model and skill selection propagation into run creation;
- thread row status badges;
- run companion tab badges and auto-focus rules;
- responsive collapse behavior where practical.

Manual browser verification should cover:

- desktop three-column layout;
- collapsed left rail;
- collapsed right companion;
- mobile/narrow layout;
- running, waiting input, approval needed, failed, and completed states.

## Out of Scope

This design does not add:

- an agent workflow graph editor;
- persisted AgentPlan-as-workflow definitions;
- workflow scheduling;
- drag-and-drop workflow nodes or edges;
- unrestricted model tool execution outside the capability boundary.

## Open Implementation Decisions

These can be decided during planning:

- exact tab auto-focus rules for Approvals and Files;
- whether Activity rows are derived directly from stream events or from an intermediate
  projection helper;
- how much usage information appears in the transcript versus the right panel;
- whether files/artifacts share one tab initially or split later.
